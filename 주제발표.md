## 슬라이드 3 — 왜 이 주제인가 (배경)

**핵심 메시지**: KABONE 평가는 단계 누락만으로 떨어지지만, 학생이 집에서 평가 기준 그대로 반복할 수단이 없다.

**본문**
- KABONE 핵심기본간호술 평가는 **단계 누락 또는 핵심항목 통과 실패 시 통과가 어렵다**.
- 그러나 학생이 집에서 평가 기준 그대로 반복 연습할 수단은 **교과서와 영상뿐**이다.
- 영상은 관찰 학습은 되지만 **의사결정 훈련**(어떤 바늘을 고를지, 어느 랜드마크를 찾을지)은 불가능하다.
- 학교 마네킹은 비싸고, 실습실 시간은 제한적이며, 학교 밖 접근이 불가하다.
- 해외 제품(Practigame 등)은 **한국어 미지원** + KABONE 평가 기준과 불일치.

**발표 멘트**
> "왜 이 주제냐고 한다면, KABONE 평가는 한 단계만 빠뜨려도 떨어질 수 있는 평가인데도 학생이 집에서 똑같이 반복할 도구가 없다는 점입니다. 영상은 보기만 가능하고, 마네킹은 학교 밖에선 못 만지고, 해외 시뮬레이터는 한국어도 KABONE 기준도 맞지 않습니다."

**출처**: `docs/01-product-vision.md` §해결하려는 문제

## 슬라이드 8 — 시스템 아키텍처

**핵심 메시지**: 데이터 드리븐 — 시나리오는 코드가 아니라 ScriptableObject 자산이다.

**본문**

```
NursingScenario (SO)
  └─ List<ScenarioStep>  (추상 SO, 7개 구체 타입)
       ChecklistStep, ToolInteractionStep, DialogueStep,
       SelectionStep, LandmarkPickStep, SequenceStep, ToggleGroupStep

런타임:
[Player Input] → InteractionManager (Raycast)
                  ↓
                IInteractable
                  ↓
              IStepController (7종)
                  ↓
              FeedbackBus (SO 이벤트 6채널)
                  ↓
       ┌──────┬─────────┬───────────┬──────────┐
      UI    Audio    Scoring     SaveService   …
```

- **데이터 드리븐**: 시나리오 추가/수정 = `.asset` 편집, 코드 수정 불요
- **FSM**: ScenarioRunner가 Idle → StepActive → StepEvaluating → Completed 순환
- **느슨한 결합**: FeedbackBus의 SO 이벤트 채널로 UI/오디오/채점/저장이 독립 구독

**발표 멘트**
> "아키텍처의 핵심은 두 가지입니다. 첫째, 시나리오 자체가 코드가 아니라 ScriptableObject 자산입니다. 둘째, 모든 컴포넌트가 ScriptableObject 이벤트 채널인 FeedbackBus로 통신하기 때문에 채점·UI·오디오·저장이 서로 모르고도 동작합니다. 이 구조를 택한 이유는 시나리오 확장 시 코드 수정 없이 자산 편집만으로 늘릴 수 있도록 하기 위함입니다."

**출처**: `docs/03-technical-spec.md` §아키텍처, `docs/05-data-model.md` §1~2

## 슬라이드 9 — 채점 · 평가 시스템

**핵심 메시지**: 100점 가중치 + 6개 핵심 게이트 + 명시적 감점 사유 enum.

**본文**
- **총점**: 100점, 단계별 가중치 분배
  - 핵심항목(★) 6개 = 60점
  - 안전(손위생 4회 + 소독 + 폐기) = 25점
  - 소통·문서화 = 15점
- **Critical fail**: 핵심항목 통과 실패 시 해당 단계 0점 + 디브리핑 경고 배지
- **`DeductionReason` 감점 사유 enum** (예시)
  - `HandHygieneSkipped` — 손위생 누락
  - `DoseOutOfTolerance` — 용량 오차 ±0.1ml 초과
  - `ClosedQuestionOnly` — 폐쇄형 질문만으로 환자 식별
  - `OneIdentifierOnly` — 식별자 1개만 사용
  - `AngleOutOfRange` — 주사 각도 범위 이탈 (90°±10°)
  - `AspirationSkipped` — 흡인 단계 누락
  - `BloodSeenButContinued` — 혈액 역류 후 그대로 주입
  - `SharpsBinMissed` — 손상성 폐기물 분리 실패
  - … 등 20여 종

## 부록 B — 자주 받을 만한 질문 + 답안

**Q1. 왜 IM 주사 1종만 합니까? 너무 좁지 않나요?**
> 학기 12주 일정 안에서 KABONE 21단계를 깊이 있게 구현하는 것이 우선입니다. 1종 고정은 의도된 스코프 결정이며, 같은 아키텍처로 다른 술기를 추가할 수 있도록 시나리오를 ScriptableObject 자산으로 만듭니다.

**Q2. 의학 지식은 어떻게 검증하나요?**
> 별도 감수자 컨택 없이 KABONE이 발간한 공식 평가항목 프로토콜 제4.1판 PDF를 1차 출처로 직접 인용합니다. 모든 절차 줄에 KABONE 단계 ID가 붙어 추적 가능합니다.

**Q3. 모바일은 안 합니까?**
> 학교 단위 PC 배포가 1차 목표라 MVP는 Windows + macOS입니다. 모바일은 입력·UI 재설계가 필요해 MVP 이후로 미뤘습니다.

**Q4. 컴퓨터그래픽스 수업 학습 요소가 어떻게 들어갑니까?**
> 슬라이드 6 표 그대로입니다 — Cinemachine, New Input System, Raycast 인터랙션, ScriptableObject 이벤트, 싱글턴, 애니메이션, 포스트 프로세싱이 한 도메인 안에서 모두 사용됩니다.

**Q5. 디지털트윈인 게 맞나요?**
> 실제 처치실의 공간·환자·도구·절차를 디지털 대응체로 만들고, 학생이 그 안에서 평가 기준에 정렬된 의사결정을 반복하는 구조입니다. 광의의 디지털트윈 정의에 부합합니다.

**Q6. Unity 어셋스토어에서 다 사 오면 되는 거 아닌가요?**
> 환경·캐릭터는 Microsoft Rocketbox(MIT) 같은 무료 자원을 활용하지만, 시뮬레이터의 핵심 가치 — KABONE 21단계 절차 매핑·채점·디브리핑 — 는 직접 설계·구현해야 하는 부분입니다.
